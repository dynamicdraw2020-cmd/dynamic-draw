# 관리자 사용 설명서

## 1. 관리자 로그인

일반 로그인 화면에서 관리자 이메일과 비밀번호를 입력한다. 승인된 관리자 역할이면 `/admin` 대시보드로 이동한다.

역할은 VIEWER, MANAGER, SUPER_ADMIN 순으로 권한이 커진다. 최초 최고 관리자는 배포 직후 `https://배포주소/setup-admin`에서 만든다. Vercel의 `ADMIN_SETUP_SECRET` 값을 입력해 한 번 생성하면 설치 화면은 자동으로 잠긴다. `supabase/MAKE_FIRST_ADMIN.sql`은 설치 화면이 동작하지 않을 때만 사용하는 예비 방법이다.

비밀번호를 잊었다면 일반 회원과 같은 **비밀번호를 잊었나요?** 흐름을 사용한다. 관리자 권한은 이메일 링크만으로 새로 생기지 않으며 기존 DB 역할이 유지된다.

## 2. 역할별 할 수 있는 일

| 역할 | 할 수 있는 일 |
|---|---|
| VIEWER | 대시보드, 결과, 통계, 확률 기록, 관리자 로그 조회 |
| MANAGER | VIEWER 기능 + 회원 처리, 뽑기·상품·확률·교환 규칙 관리, 추첨 실행 |
| SUPER_ADMIN | MANAGER 기능 + 결과 무효, 사이트 설정, 회원 역할 변경 |

모든 관리자 계정은 상태가 `APPROVED`여야 한다. 권한이 있어도 정지 상태면 관리자 기능을 사용할 수 없다.

## 3. 대시보드

누적 추첨, 오늘 추첨, 가입 승인 대기 수, 진행 중인 뽑기 수, 최근 결과, 최근 관리자 활동을 확인한다. 현장 운영 전 승인 대기와 진행 상태를 먼저 본다.

## 4. 회원 승인과 고유 ID 발급

1. **회원 관리**로 이동한다.
2. 이름·이메일·연락처·고유 ID 검색창으로 대상을 확인한다.
3. `승인 대기` 회원의 신청 정보를 확인한다.
4. **승인·ID 자동발급**을 누른다.
5. 확인 창에서 승인한다.
6. 서버가 `DD-연도-6자리번호`를 발급한 결과를 확인한다.

고유 ID는 PostgreSQL 시퀀스와 중복 검사를 통해 서버에서 생성된다. 중복 ID는 저장되지 않는다. 반려 또는 정지할 때는 반드시 사유를 입력한다. 승인·반려·정지·복구 행동은 관리자 로그에 남는다.

### 회원 상태 의미

- `PENDING`: 신청 후 승인 전
- `APPROVED`: 정상 이용 가능
- `REJECTED`: 신청 반려
- `SUSPENDED`: 기존 계정 이용 정지

현재 로그인한 관리자 자신은 회원 처리 버튼으로 정지할 수 없다.

## 5. 관리자 역할 지정

이 기능은 SUPER_ADMIN만 사용한다.

1. **회원 관리**에서 승인된 회원을 찾는다.
2. 권한 칸에서 `일반 회원`, `조회 관리자`, `일반 관리자`, `최고 관리자` 중 하나를 고른다.
3. **저장**을 누르고 확인 창을 승인한다.
4. 대상 사용자는 다음 요청 또는 다시 로그인할 때 새 권한이 적용된다.

안전 장치:

- 승인된 회원만 관리자로 올릴 수 있다.
- 현재 로그인한 최고 관리자는 자기 권한을 직접 낮출 수 없다.
- 마지막 남은 최고 관리자의 권한은 낮출 수 없다.
- 모든 역할 변경은 `MEMBER_ROLE_CHANGED` 관리자 로그에 남는다.

권장 운영은 최고 관리자 2명, 일반 관리자 필요한 인원, 조회 관리자는 통계·감사 확인 담당으로 구성하는 것이다.

## 6. 뽑기 만들기

1. **뽑기·상품·확률 관리**로 이동한다.
2. 뽑기 이름, 설명, 연출 시간 3~5초를 입력한다.
3. **뽑기 생성**을 누른다.
4. 상품을 하나씩 추가한다.
5. 상품별 확률을 입력한다.
6. 변경 사유를 쓰고 확률을 저장한다.
7. 합계가 100%이고 재고 문제가 없으면 **시작**을 누른다.

### 뽑기 상태

- **초안(DRAFT):** 관리자 준비 단계
- **진행(ACTIVE):** 추첨 가능
- **일시정지(PAUSED):** 잠시 추첨 중단
- **종료(ENDED):** 행사 종료

공개 설정을 끈 초안·비공개 뽑기는 공개 목록과 공개 통계에 나타나지 않는다.

### 상품 속성

- **회원 보유 상품:** 결과가 공개될 때 회원 보유함에 수량이 들어간다.
- **교환 재료:** 교환 규칙의 차감 상품으로 쓰기 좋은 항목이다.
- **재고:** 비워 두면 무제한이다. 숫자를 넣으면 당첨 결과 확정 또는 교환 지급 때 차감된다.
- **비활성화:** 과거 기록은 남기고 앞으로 사용하지 않는다. 먼저 확률을 0%로 저장해야 한다.

이미 회원이 보유하거나 결과·교환 규칙에 사용 중인 상품은 기록 보호를 위해 완전 삭제하지 않는다. 비활성화로 운영한다.

## 7. 확률 관리

확률은 퍼센트 소수점 네 자리까지 입력할 수 있다. 내부에서는 100%를 1,000,000 정수 단위로 저장하므로 부동소수점 오차 없이 합계를 검사한다.

저장 조건:

- 모든 활성 상품을 한 번씩 입력
- 값이 0 이상
- 합계가 정확히 100%
- 변경 사유 2자 이상

확률 0% 상품은 누적 구간 경계와 관계없이 선택 대상에서 제외된다. 저장되면 변경 전 값, 변경 후 값, 관리자, 뽑기, 사유, IP, User-Agent, 시각, 이전 해시, 현재 해시가 기록된다. 해당 행은 수정·삭제할 수 없다.

## 8. 실시간 추첨 실행

1. **실시간 추첨**으로 이동한다.
2. 진행 중인 뽑기를 고른다.
3. 승인된 일반 회원을 고른다.
4. 회원 고유 ID를 다시 확인한다.
5. **추첨 실행**을 한 번 누른다.

서버는 뽑기 상태, 회원 상태, 확률 합계, 재고, 진행 중인 이전 연출을 검사한다. 성공하면 결과와 확률 스냅샷을 DB에 먼저 확정하고 공개 화면에 시작 이벤트를 보낸다. 상품명은 공개 시각까지 공개 이벤트에 넣지 않는다. 3~5초 뒤 결과가 공개되는 순간 보관 상품이 회원 보유함에 한 번 적립된다.

동일 버튼 요청에는 고유 멱등 키가 붙는다. 같은 요청이 재전송되어도 결과와 적립은 한 번만 처리된다. 한 뽑기에 공개되지 않은 이전 결과가 있으면 다음 추첨을 막는다.

### 현장 운영 권장 순서

- 공개 화면을 별도 모니터에 `/live`로 띄운다.
- 관리자 화면은 다른 기기 또는 다른 브라우저 창에서 연다.
- 참가자의 이름과 고유 ID를 구두로 확인한다.
- 추첨 버튼을 한 번만 누르고 결과 공개 문구가 나올 때까지 기다린다.
- 공개 뒤 회원 보유함 또는 결과 목록이 갱신되었는지 확인한다.

## 9. 교환 규칙과 관리자 현장 교환

### 교환 규칙 만들기

1. **교환 시스템**으로 이동한다.
2. 규칙 이름을 입력한다.
3. 차감 상품과 수량을 고른다.
4. 지급 상품과 수량을 고른다.
5. 규칙을 추가한다.

찢어진 입장권 외의 어떤 보유 상품도 사용할 수 있다. 차감 상품과 지급 상품은 달라야 한다. 수량은 1 이상이어야 한다. 유한 재고 지급 상품은 재고가 있어야 교환된다. 규칙을 끄거나 비활성화해도 기존 교환 기록은 남는다.

### 고유 ID로 현장 교환하기

1. **교환 시스템** 상단의 **관리자 현장 교환**을 찾는다.
2. 회원에게 고유 ID를 확인한다.
3. `DD-연도-번호`를 입력한다. 입력칸의 추천 목록에서 회원을 선택할 수도 있다.
4. 적용할 교환 규칙을 고른다.
5. **현장 교환 실행**을 누른다.
6. 회원 이름·ID와 차감·지급 수량을 다시 확인한다.
7. 성공 안내를 확인한다.

보유 수량이 부족하거나 대상 상품 재고가 없으면 교환은 전혀 반영되지 않는다. 관리자가 대신 실행한 교환은 `ADMIN_EXCHANGE_EXECUTED` 감사 로그로 남는다.

## 10. 결과 관리

결과는 완전 삭제하지 않는다. 문제가 있는 공개 결과는 SUPER_ADMIN이 사유를 입력해 **무효 처리**한다.

무효 처리 시:

- 회원 보유함에 적립된 상품 1개를 회수한다.
- 유한 재고 상품이면 재고를 되돌린다.
- 공개 목록과 통계에서 제외한다.
- 무효 시각, 사유, 처리 관리자를 남긴다.
- 원 결과 행과 감사 로그는 보존한다.

회원이 이미 해당 상품을 교환·사용해 보유 수량이 0이면 무효 처리가 거부된다. 먼저 교환 내역을 확인하고 운영 정책에 따라 처리한다. 아직 공개되지 않은 결과는 일반 무효 처리 대상이 아니며 공개 복구 상태를 먼저 확인한다.

## 11. 통계와 기록

- **통계:** 상품별 횟수, 실제 출현율, 설정 확률 비교, 최근 7일 추첨 수
- **확률 변경 기록:** 뽑기·관리자·시작일·종료일 필터, 변경 전후 JSON, 사유, IP, 해시 체인 검증 결과
- **관리자 로그:** 로그인, 뽑기·상품·회원·역할·교환·설정 변경, 추첨, 결과 공개·무효, 해시 체인 검증 결과

감사 로그의 `entry_hash`는 직전 행의 해시를 포함한다. DB는 행 수정·삭제를 차단하며, 관리자 화면은 저장된 모든 행의 해시를 다시 계산해 **정상** 또는 **이상 감지**를 표시한다.

`이상 감지`가 나오면 새 추첨·확률 변경을 잠시 중단하고 Supabase 원본과 최근 관리자 작업을 점검한다.

## 12. 사이트 설정

SUPER_ADMIN은 다음 항목을 바꿀 수 있다.

- 사이트 이름
- 메인 제목
- 메인 설명
- 공개 통계 켜기/끄기

설정 변경도 관리자 로그에 남는다. 행사 중 큰 문구 변경은 공개 화면을 확인한 뒤 진행한다.

## 13. 운영 전 체크리스트

- [ ] 승인할 회원의 신원을 확인했다.
- [ ] 관리자 역할과 담당 업무를 확인했다.
- [ ] 진행 중인 뽑기가 맞다.
- [ ] 확률 합계가 100%다.
- [ ] 확률이 있는 모든 유한 재고 상품의 재고가 1개 이상이다.
- [ ] 공개 `/live` 화면의 LIVE 표시가 보인다.
- [ ] 테스트 회원으로 1회 추첨과 교환을 확인했다.
- [ ] 결과 공개 뒤 보유함 적립을 확인했다.
- [ ] 확률 기록과 관리자 로그의 해시 체인이 정상이다.
- [ ] 관리자 비밀키가 GitHub에 올라가지 않았다.
- [ ] 행사 종료 후 뽑기를 `종료` 상태로 바꾼다.
