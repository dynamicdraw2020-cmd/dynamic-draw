# Dynamic Draw 서비스 기획서

문서 버전: 1.2  
기준일: 2026-06-24

## 1. 서비스 정의

Dynamic Draw는 이벤트 운영자가 회원별 추첨을 실행하고, 일반 사용자가 확률·과정·결과·통계를 확인하는 결제 없는 웹 기반 추첨 서비스다. 결과 결정은 브라우저가 아니라 PostgreSQL 트랜잭션 안에서 수행되며, 카드 애니메이션은 이미 확정된 결과를 공개 시점에 맞춰 보여 주는 연출만 담당한다.

## 2. 핵심 목표

1. 운영자가 행사 현장에서 승인 회원을 빠르게 선택해 추첨할 수 있어야 한다.
2. 이용자는 확률 합계, 최근 결과, 설정 확률과 실제 출현율을 확인할 수 있어야 한다.
3. 확률 변경과 관리자 행동은 누가 언제 왜 바꾸었는지 추적할 수 있어야 한다.
4. 찢어진 입장권뿐 아니라 관리자가 지정한 어떤 보유 상품도 다른 상품으로 교환할 수 있어야 한다.
5. 회원가입은 신청형으로 운영하고, 관리자가 승인할 때 고유 회원 ID를 발급해야 한다.
6. 개인 개발자 한 명이 Vercel과 Supabase 조합으로 운영·확장할 수 있어야 한다.

## 3. 사용자와 권한

### 일반 방문자

로그인하지 않고 메인, 진행 뽑기, 확률표, 실시간 연출, 최근 결과, 공개 통계를 확인한다.

### 일반 회원

회원가입 신청, 이메일 인증, 승인 상태 확인, 로그인, 비밀번호 재설정, 고유 ID 확인, 개인 보유 상품 확인, 교환, 본인 추첨·교환 기록 확인이 가능하다.

### VIEWER 관리자

대시보드, 결과, 통계, 확률 변경 기록, 관리자 로그를 조회한다. 데이터를 바꾸지는 못한다.

### MANAGER 관리자

VIEWER 권한에 더해 회원 승인·반려·정지·복구, 뽑기·상품·확률·교환 규칙 관리, 실시간 추첨 실행이 가능하다.

### SUPER_ADMIN

MANAGER 권한에 더해 결과 무효 처리, 사이트 전역 설정, 회원 역할 변경이 가능하다. 마지막 남은 최고 관리자의 권한을 낮추는 동작은 차단한다.

## 4. 주요 업무 흐름

### 회원가입과 승인

1. 이용자가 이름 또는 닉네임, 연락처, 이메일, 비밀번호를 입력해 신청한다.
2. Supabase Auth가 계정을 만들고 DB 트리거가 `profiles`에 `PENDING` 회원을 만든다.
3. 이메일 확인 기능이 켜져 있으면 이용자가 받은 인증 링크를 누른다.
4. 관리자가 신청 내용을 확인한다.
5. 관리자가 승인하면 서버가 `DD-연도-6자리번호` 형식의 고유 ID를 자동 발급한다.
6. 회원은 다시 로그인해 개인 페이지와 교환 기능을 사용한다.

### 비밀번호 재설정

1. 이용자가 로그인 화면에서 **비밀번호를 잊었나요?**를 누른다.
2. 가입 이메일을 입력하면 Supabase가 복구 링크를 보낸다.
3. 이용자가 링크를 누르고 새 비밀번호를 저장한다.
4. 승인 상태와 역할은 바뀌지 않고 비밀번호만 교체된다.

### 추첨

1. 관리자가 진행 중인 뽑기와 승인된 일반 회원을 선택한다.
2. 서버가 관리자 세션, 승인 상태, 권한, 동일 출처 요청을 검사한다.
3. DB가 뽑기, 회원, 확률, 상품 재고를 잠금 상태에서 검증한다.
4. `pgcrypto` 보안 난수와 rejection sampling으로 0~999,999 값을 만들고, 확률이 0보다 큰 상품의 누적 확률 구간에서 하나를 선택한다.
5. 결과·확률 스냅샷·재고 차감을 한 트랜잭션으로 먼저 확정한다. 이 시점에는 상품명을 공개하지 않는다.
6. `DRAW_START`, `DRAW_ANIMATING` 이벤트를 보내 3~5초 카드 연출을 시작한다.
7. 공개 시각이 되면 결과를 공개하고, 보관 상품이면 회원 보유함에 한 번만 적립한다.
8. `DRAW_RESULT`, `STATS_UPDATE` 이벤트를 보내 공개 화면과 통계를 새로고침한다.

### 교환

1. 회원이 교환 탭에서 현재 보유 수량과 활성 규칙을 확인한다.
2. 서버가 회원 본인 여부와 승인 상태를 검사한다.
3. DB가 회원·규칙·상품·보유 수량을 잠근다.
4. 재료 차감, 대상 지급, 유한 재고 차감, 교환 로그 저장을 한 번에 처리한다.
5. 동일 요청 ID가 다시 전송되어도 한 번만 처리한다.

### 결과 무효 처리

1. 결과는 완전 삭제하지 않는다.
2. SUPER_ADMIN이 사유를 입력해 무효 처리한다.
3. 이미 적립된 보관 상품을 회수하고 유한 재고를 복구한다.
4. 공개 목록과 통계에서는 제외하지만 원 결과와 감사 기록은 남긴다.

## 6. UI 구조

### 공개 화면

```text
상단 고정 메뉴
├─ 로고
├─ 진행 중인 뽑기
├─ 실시간 결과
├─ 확률표
├─ 최근 당첨
├─ 통계
└─ 로그인 / 내 정보

메인
├─ 서비스 소개와 행동 버튼
├─ 누적 지표
├─ 진행 상품 카드
├─ 실시간 카드 연출
├─ 최근 결과
└─ 투명성 안내
```

### 회원 화면

```text
내 정보
├─ 고유 회원 ID와 승인 상태
├─ 보유 상품 수량
├─ 내 최근 추첨 결과
└─ 내 교환 기록

교환
├─ 현재 재료 보유 수량
├─ 활성 교환 규칙 카드
└─ 교환 실행·성공/실패 안내
```

### 관리자 화면

```text
관리자 공통 레이아웃
├─ 대시보드
├─ 뽑기·상품·확률
├─ 실시간 추첨
├─ 회원·역할 관리
├─ 교환 시스템
├─ 결과 관리
├─ 통계
├─ 확률 변경 기록·체인 검증
├─ 관리자 로그·체인 검증
└─ 설정
```

모바일에서는 공개 메뉴가 햄버거 메뉴로, 관리자 메뉴가 가로 스크롤 메뉴로 바뀐다. 넓은 표는 가로 스크롤하고, 입력 폼은 한 열로 배치한다.

## 7. 비기능 요구사항

- 화면 날짜는 한국 표준시로 표시한다.
- 최소 권장 너비 390px부터 핵심 기능을 사용할 수 있어야 한다.
- 관리자 변경 API는 인증·권한·승인 상태·동일 출처 요청을 검사한다.
- 가입·로그인·비밀번호 복구·추첨·교환·공개 복구 요청에 횟수 제한을 둔다.
- 추첨과 교환은 고유 요청 키로 중복 버튼 클릭에도 한 번만 반영한다.
- RLS와 컬럼 권한으로 브라우저의 직접 DB 접근 범위를 제한한다.
- 확률 기록과 감사 로그의 UPDATE·DELETE를 DB 트리거로 차단한다.
- 해시 체인을 관리자 화면에서 다시 계산해 이상 여부를 표시한다.
- 결과 공개 지연 중 관리자 화면이 닫혀도 공개 화면의 복구 호출로 자동 공개한다.
- 비밀키는 GitHub나 브라우저 번들에 포함하지 않고 Vercel 서버 환경변수에만 둔다.

## 8. 기본 운영 정책

- 공개 화면의 이름과 고유 ID는 일부 가려서 표시한다.
- 초안 또는 비공개 뽑기는 공개 통계에 포함하지 않는다.
- 확률 0% 상품은 경계값과 관계없이 절대 선택하지 않는다.
- 보관 상품 적립은 결과가 실제 공개되는 순간 한 번만 실행한다.
- 결과는 삭제하지 않고 무효 처리한다.
- 회원 역할 변경은 SUPER_ADMIN만 가능하다.
- 최고 관리자는 최소 한 명을 유지한다.

## 9. 개발 로드맵

### 현재 v1.1

회원 승인과 고유 ID 자동 발급, 첫 관리자 설치 페이지, 비밀번호 재설정, 역할 관리, 뽑기, 확률, 실시간 공개, 자동 적립, 회원 직접·관리자 현장 교환, 기록 필터, 공개·관리자 통계, 감사 로그와 해시 체인 검증, 반응형 화면까지 포함한다.

### v1.2

CSV 회원 일괄 등록, CSV 결과·교환 다운로드, 상품 이미지 업로드, 관리자 검색·필터·페이지네이션, 승인·반려 이메일 알림을 추가한다.

### v1.3

QR 회원 확인, 이벤트별 공개 URL, Discord·카카오 알림, 시즌별 통계, 현장 전체 화면과 효과음을 추가한다.

### v2

추첨 무결성 리포트, 외부 검증용 결과 커밋, 외부 저장소 감사 해시 앵커, 복수 행사 조직과 세분화된 권한, 대규모 Realtime Broadcast 구조를 도입한다.
