# Dynamic Draw 운영 체크리스트와 문제 해결

문서 버전: 1.0  
기준일: 2026-06-24

이 문서는 홈페이지를 올린 다음 실제 행사에서 안전하게 쓰기 위한 운영 순서와 오류 대처 방법을 모아 둔 문서다.

# 1. 첫 배포가 끝난 직후 확인

## 1.1 접속 확인

- [ ] Vercel 임시 주소가 열린다.
- [ ] `https://dynamic2020.com`이 열린다.
- [ ] `https://www.dynamic2020.com`도 열리거나 루트 주소로 이동한다.
- [ ] 브라우저 주소창에 자물쇠가 보인다.
- [ ] 휴대전화 Wi-Fi와 모바일 데이터에서 각각 열린다.

## 1.2 Supabase 연결 확인

- [ ] `/api/health`를 열었을 때 production 모드와 Supabase 연결 상태가 정상이다.
- [ ] 가입 신청이 DB `profiles`에 `PENDING`으로 생긴다.
- [ ] 이메일 인증 링크가 올바른 사이트로 돌아온다.
- [ ] 첫 최고 관리자 계정으로 `/admin`에 들어간다.
- [ ] 관리자 로그에 로그인 기록이 남는다.

## 1.3 기본 데이터 확인

- [ ] 기본 `입장권 뽑기`가 보인다.
- [ ] `꽝`, `찢어진 입장권`, `DwX`, `Dynamic` 상품이 보인다.
- [ ] 기본 확률은 55%, 40%, 4%, 1%이고 합계가 100%다.
- [ ] 찢어진 입장권 5개 → DwX, 8개 → Dynamic 규칙이 보인다.

# 2. 행사 하루 전 준비

1. 운영자별로 개인 관리자 계정을 만든다.
2. 최고 관리자는 필요한 계정에 VIEWER 또는 MANAGER 역할을 준다.
3. 공용 관리자 비밀번호 하나를 여러 명이 같이 쓰지 않는다.
4. 테스트 회원 2명을 승인하고 고유 ID를 확인한다.
5. 테스트 뽑기 또는 실제 뽑기에서 1회 추첨한다.
6. 공개 모니터 `/live`에서 카드 연출과 결과를 확인한다.
7. 결과 공개 뒤 테스트 회원 보유함에 상품이 들어왔는지 확인한다.
8. 재료 수량을 확보한 테스트 계정으로 교환을 확인한다.
9. 확률 변경 기록과 관리자 로그에서 해시 체인 **정상**을 확인한다.
10. 행사장 인터넷과 예비 휴대전화 핫스팟을 준비한다.

# 3. 행사 시작 30분 전

## 공개용 기기

1. 브라우저에서 `https://dynamic2020.com/live`를 연다.
2. 전체 화면 모드로 전환한다.
3. LIVE 표시와 최근 결과가 보이는지 확인한다.
4. 화면 절전 시간을 길게 바꾼다.
5. 공개용 기기에는 관리자 계정으로 로그인하지 않는다.

## 관리자용 기기

1. 개인 관리자 계정으로 로그인한다.
2. 올바른 뽑기가 `진행` 상태인지 확인한다.
3. 회원 목록과 고유 ID를 확인한다.
4. 상품 재고와 확률 합계 100%를 확인한다.
5. 브라우저 새로고침 후에도 로그인이 유지되는지 확인한다.

# 4. 실제 추첨 한 번의 표준 순서

1. 참가자에게 이름과 고유 ID를 확인한다.
2. 관리자 화면에서 같은 회원을 선택한다.
3. 공개 모니터가 대기 상태인지 확인한다.
4. **추첨 실행**을 한 번 누른다.
5. 카드가 흔들리고 빛나고 뒤집힐 때까지 다른 버튼을 누르지 않는다.
6. 공개 모니터와 관리자 결과가 같은지 확인한다.
7. 보관 상품이면 회원 보유함에 적립되었는지 확인한다.
8. 다음 참가자로 넘어간다.

# 5. 일반 회원 문의에 답하는 법

## “가입했는데 교환이 안 돼요”

회원 관리에서 상태를 확인한다.

- PENDING: 승인과 고유 ID 발급 필요
- REJECTED: 반려 사유 확인
- SUSPENDED: 정지 사유 확인
- APPROVED: 보유 수량과 활성 교환 규칙 확인

## “비밀번호를 잊었어요”

로그인 화면의 **비밀번호를 잊었나요?**를 누르라고 안내한다. 관리자가 회원 비밀번호를 직접 보거나 알려 주면 안 된다.

## “상품이 안 들어왔어요”

1. 결과가 실제로 공개되었는지 확인한다.
2. 상품이 `회원 보유 상품`으로 설정되어 있는지 확인한다.
3. 결과가 무효 처리되지 않았는지 본다.
4. 참가자 고유 ID가 맞는지 확인한다.
5. Supabase의 `results`, `participant_items`, `admin_logs`를 점검한다.

## “교환 버튼이 꺼져 있어요”

재료 수량, 규칙 활성 상태, 대상 상품 활성 상태, 유한 재고를 확인한다.

# 6. 자주 발생하는 배포 오류

## Vercel에서 빌드가 실패함

확인 순서:

1. GitHub 저장소 첫 화면에 `package.json`이 바로 보이는지 확인한다.
2. ZIP 파일 하나만 올린 것이 아닌지 확인한다.
3. Vercel Build Logs에서 가장 아래쪽 첫 빨간 오류를 본다.
4. 환경변수 이름의 철자와 앞뒤 공백을 확인한다.
5. 코드를 수정한 뒤 GitHub에 다시 저장하면 새 배포가 시작된다.

## 홈페이지는 열리지만 “미리보기 모드”라고 나옴

여섯 환경변수 중 하나가 없거나 `NEXT_PUBLIC_DEMO_MODE=true`인 상태다.

```text
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY
SUPABASE_SECRET_KEY
NEXT_PUBLIC_SITE_URL
NEXT_PUBLIC_DEMO_MODE=false
ADMIN_SETUP_SECRET
```

Vercel에서 값을 고친 뒤 반드시 Redeploy한다.

## 가입은 되는데 이메일 링크가 엉뚱한 곳으로 감

Supabase `Authentication → URL Configuration`에서 Site URL과 Redirect URLs를 확인한다.

```text
https://dynamic2020.com
https://dynamic2020.com/**
https://www.dynamic2020.com/**
https://프로젝트이름.vercel.app/**
```

## 관리자 페이지에 못 들어감

1. Vercel 배포 주소 뒤에 `/setup-admin`을 붙여 최초 최고 관리자를 만들었는지 확인한다.
2. 설치 화면에 “DB 설치 SQL이 아직 실행되지 않았습니다”가 나오면 `PASTE_THIS_ONCE.sql` 실행 여부를 확인한다.
3. 설치 화면에 “ADMIN_SETUP_SECRET이 없거나 짧습니다”가 나오면 Vercel 환경변수를 고치고 Redeploy한다.
4. 관리자 생성에 성공했다면 `/login`에서 같은 이메일과 비밀번호로 로그인한다.
5. 그래도 안 되면 Supabase `profiles`에서 `role=SUPER_ADMIN`, `status=APPROVED`인지 확인한다.
6. `MAKE_FIRST_ADMIN.sql`은 위 설치 화면을 사용할 수 없을 때만 쓰는 예비 방법이다.

## 실시간 화면이 움직이지 않음

1. 공개 화면을 새로고침한다.
2. 뽑기가 `ACTIVE`인지 확인한다.
3. Supabase Realtime publication에 `live_events`가 포함됐는지 확인한다. 설치 SQL이 이를 설정한다.
4. 브라우저 개발자 도구의 네트워크 차단 확장 프로그램을 잠시 끈다.
5. `/api/live/reveal-due`가 오류인지 확인한다.

## 도메인이 안 열림

1. Vercel Domains 화면의 정확한 A/CNAME 값을 다시 본다.
2. 가비아 DNS에서 `@`와 `www` 값을 비교한다.
3. CNAME 값 마지막에 점이 필요한지 확인한다.
4. 기존 `@` 또는 `www` 웹 연결 레코드가 충돌하는지 본다.
5. MX, SPF, DKIM 같은 이메일 레코드는 지우지 않는다.
6. DNS는 반영 시간이 필요할 수 있으므로 Vercel의 검증 상태를 주기적으로 새로고침한다.

# 7. 운영 중 긴급 상황

## 잘못된 회원을 선택해 추첨함

1. 다음 추첨을 잠시 멈춘다.
2. 결과가 공개될 때까지 상태를 확인한다.
3. SUPER_ADMIN이 결과 관리에서 구체적인 사유로 무효 처리한다.
4. 상품 회수 실패가 나오면 해당 회원의 교환 내역을 먼저 확인한다.
5. 올바른 회원으로 다시 추첨한다.
6. 관리자 로그에 두 작업이 모두 남았는지 확인한다.

## 확률을 잘못 저장함

1. 해당 뽑기를 `일시정지`로 바꾼다.
2. 올바른 확률을 입력하고 구체적인 변경 사유를 남긴다.
3. 확률 변경 기록의 전후 값과 해시 체인 정상 여부를 확인한다.
4. 이미 실행된 결과는 당시 확률 스냅샷으로 보존된다.

## 감사 해시 체인 “이상 감지”가 나옴

1. 새 추첨과 확률 변경을 잠시 중단한다.
2. 이상이 표시된 순번과 시각을 기록한다.
3. 최근 관리자 작업과 Supabase 로그를 확인한다.
4. DB 백업을 만든 뒤 원인을 조사한다.
5. 단순히 화면 표시를 없애려고 감사 행을 수정하거나 삭제하지 않는다.

## Secret key가 외부에 노출된 것 같음

1. Supabase에서 비밀키를 즉시 회전한다.
2. Vercel의 `SUPABASE_SECRET_KEY`를 새 값으로 바꾼다.
3. Redeploy한다.
4. GitHub 기록에 키가 들어갔다면 파일 삭제만으로 끝내지 말고 키를 반드시 폐기한다.
5. 관리자 로그와 Supabase 로그에서 수상한 접근을 확인한다.

# 8. 백업과 개인정보

- 행사 전과 행사 종료 뒤 Supabase 데이터베이스 백업 정책을 확인한다.
- 결과·확률·감사 기록은 운영 정책에 맞춰 보존한다.
- 이름, 이메일, 연락처는 개인정보이므로 필요한 사람만 접근한다.
- 행사 목적이 끝난 회원 개인정보의 보존 기간과 삭제 절차를 미리 정한다.
- CSV로 내려받은 개인정보 파일은 공용 메신저나 공개 드라이브에 올리지 않는다.

# 9. 월간 점검

- [ ] 관리자 계정 중 퇴사자·불필요 계정을 정지했다.
- [ ] 최고 관리자가 최소 두 명이고 모두 개인 계정이다.
- [ ] 확률 변경 기록과 관리자 로그 해시 체인이 정상이다.
- [ ] 사용하지 않는 뽑기는 종료 상태다.
- [ ] 오래된 교환 규칙을 비활성화했다.
- [ ] 유한 재고 수량이 실제 재고와 맞다.
- [ ] Vercel 배포와 Supabase 상태에 오류가 없다.
- [ ] 노출되거나 오래된 비밀키 회전 필요성을 검토했다.
- [ ] 개인정보 보존 기간이 지난 계정을 처리했다.

# 10. 도움을 요청할 때 준비할 정보

비밀키나 비밀번호는 보내지 말고 다음만 준비한다.

- 문제가 난 화면의 전체 스크린샷
- 주소창의 URL
- 오류가 난 정확한 시각
- 눌렀던 버튼과 입력한 일반 내용
- Vercel Build Log의 빨간 오류 부분
- Supabase SQL Editor의 오류 문장
- 해당 회원의 고유 ID 또는 결과 ID 일부

이 정보가 있으면 어디에서 막혔는지 훨씬 빠르게 찾을 수 있다.
